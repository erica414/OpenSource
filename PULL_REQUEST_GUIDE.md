

# Pull Request(PR) 가이드

> 이 문서는 본 프로젝트의 GitHub 협업 규칙을 표준화하기 위한 문서이며,
> **PR 생성 → 리뷰 → 머지 → 후속 작업**까지의 전체 절차를 명확히 안내한다.
> 팀원 누구나 공통된 방식으로 작업할 수 있도록 작성되었다.

<br>

## 1. PR(Pull Request)의 개념

### 1.1 PR이란?

PR(Pull Request)은 기능 개발이 완료된 브랜치를 프로젝트의 주 브랜치(main 또는 develop)에 병합해달라고 요청하는 절차이다.    
즉, **"내가 구현한 내용을 검토 후 반영해 주세요."** 라는 공식 협업 요청이다.

### 1.2 PR의 종류
**1) 일반 PR**
> 리뷰 가능, 머지할 준비 완료한 상태

**2) Draft PR**
> “리뷰 불가” 표시, 머지 버튼 비활성화, 제목 앞에 [Draft] 표시, 리뷰어가 코멘트는 달 수 있음(리뷰 제출은 X)
* 언제 사용하나요?
  * 아직 미완성이지만 코드가 커질 것 같을 때
  * 중간 점검을 받고 싶을 때
  * 리뷰어에게 “참고용인데 보긴 해야 함”이라고 말하고 싶을 때
  * 구현 방향이 맞는지 합의하고 싶은 경우
  * 하루 작업 마무리할 때 WIP(Work In Progress)용으로 올리기

### 1.3 PR 처리 흐름 요약

| 단계        | 설명                         |
| --------- | -------------------------- |
| 기능 구현 완료  | 로컬에서 기능 구현 및 테스트           |
| 작업 브랜치 푸시 | `feature/...` 브랜치를 원격에 업로드 |
| PR 생성     | GitHub에서 PR 생성             |
| 리뷰 요청     | 리뷰어 지정 · 라벨 설정 · 본문 작성     |
| 리뷰/수정     | 피드백 반영 후 재요청               |
| 승인/머지     | Squash & Merge 권장          |
| 후속 작업     | 브랜치 삭제 · 이슈 자동 종료 확인       |

## 2. PR 생성 규칙

### 2.1 PR 생성 방법

1. GitHub → **Pull Requests → New Pull Request**
2. `base` 브랜치(병합 대상), `compare` 브랜치(작업 브랜치) 선택
- 기본 `base`브랜치가 `main`으로 선택되는 경우가 많으므로 반드시 확인
- 최종 merge의 경우 `main`, 기타 통합은 `develop`을 `base`브랜치로 선택

### 2.2 PR 제목 규칙 및 본문

* 형식: `[타입] 모듈: 설명`
* 예시
  * `[feat] login: 세션 유지 기능 추가`
  * `[fix] api: 응답 중복 문제 해결`
* 금지
  * `진짜최종`, `수정함`

> **타입 종류:** feat, fix, docs, refactor, chore, style, test, perf 등

* 본문
```md
## 변경 내용
- 

## 동작 확인 방법
1. 
2. 

## 관련 이슈
- closes #이슈번호

## 스크린샷 또는 GIF
- (필요 시 첨부)

## 비고
- 
```

### 2.3 PR 템플릿 사용하기(권장)
* GitHub repo에서 Add file → Create new file
* 파일 이름을 다음처럼 입력: `.github/pull_request_template.md`
* 템플릿 작성 후 파일 저장(commit)
* 이후 PR 만들면 자동으로 템플릿이 뜸

## 3. 리뷰(Review) 규칙

### 3.1 리뷰 요청 시

* **Reviewers**: 1~2명 할당
* **Assignees**: 작성자 또는 담당자
* **Labels**: `feature`, `bug`, `docs`, `priority:high` 등 지정

### 3.2 리뷰어 체크 항목
* [ ] 기능의 동작 및 테스트 결과 확인
* [ ] 코드 구조와 가독성
* [ ] 사이드 이펙트 가능성
* [ ] 불필요한 파일/커밋 여부
* [ ] 문서화 누락 여부

### 3.3 작성자 응답 규칙
* 모든 리뷰 코멘트에 회신
  * “수정 완료했습니다.”
  * “해당 부분은 의도적으로 유지했습니다.”
* 재수정 후에는 “Re-request Review” 클릭
* 머지 시 **Squash & Merge** 권장(히스토리 정리가 쉽고 팀 과제에 적합)

## 4. 승인 및 머지(Merge)

### 4.1 승인 조건
> PR은 기능이 100% 구현되고, 리뷰 승인 및 테스트가 통과된 경우에만 머지한다.
> 주요 기능의 develop → main 반영은 팀 회의에 따라 기능 단위 또는 주차 단위로 진행한다.

머지는 아래 조건을 모두 충족해야 한다.
* 리뷰어 최소 1명 승인(Approve)
* CI(테스트/빌드) 모두 통과
* Conflict 없음: 충돌 발생 시 작성자(리뷰어도 가능)가 GitHub 또는 IDE에서 해결 후 재요청
* PR 메타데이터 정상 작성: 제목, 본문, 라벨, 스크린샷 등 필수 항목
* base 브랜치가 올바르게 설정되어 있을 것



### 4.2 머지 전략

| 방식                 | 설명                   | 권장도     |
| ------------------ | -------------------- | ------- |
| **Squash & Merge** | 여러 커밋을 하나로 정리        | 기본 방식 |
| Rebase & Merge     | 브랜치 히스토리를 깔끔하게 정리할 때 | 선택적   |
| Merge Commit       | 릴리즈 브랜치에서 맥락 보존 필요 시 | 제한적  |

---

## 5. 머지 후(Post-Merge) 절차

1. **Delete branch** 클릭하여 브랜치 삭제
2. `closes #이슈번호`가 자동으로 작동했는지 확인: 일반적으로 main에 merge되어야 닫히므로, develop에 merge 하였을 경우 직접 닫으면 됨
3. 필요시 변경 내역을 회의록, CHANGELOG 등에 반영
4. 필요 시 태그(Release) 지정

## 6. 참고

### 체크리스트
**1) 작성자용**
* [ ] PR 제목 규칙 준수
* [ ] PR 본문 형식 준수
* [ ] 관련 이슈 연결(`closes #...`)
* [ ] 스크린샷/GIF 첨부
* [ ] 리뷰어/라벨 지정
* [ ] CI 통과
* [ ] 리뷰 반영 완료

**2) 리뷰어용**
* [ ] 기능 범위 적절
* [ ] 코드 품질·안정성 검토
* [ ] PR 설명·커밋 내용 명확
* [ ] 필요 시 수정 요청
* [ ] 승인 여부 결정


### 팁(Tips)

* **Draft PR**을 활용하면 작업 중에도 진행 상황을 팀과 공유할 수 있다.
* PR 본문은 **"무엇을 왜 변경했는가"** 중심으로 작성하면 리뷰 효율이 높아진다.
* 리뷰 코멘트는 비난이 아닌 **개선 제안**의 톤으로 작성한다.
* 기능 단위로 작은 PR을 자주 보내는 것이 협업 품질을 높인다.

### 자주 쓰는 키워드

| 구문               | 의미                  |
| ---------------- | ------------------- |
| `closes #12`     | PR 머지 시 이슈 12 자동 종료 |
| `fixes #45`      | 버그 해결               |
| `related to #34` | 관련된 이슈              |
